## Fixed Issues

### CR-01 + CR-02: Corrected-but-still-not-found serial never marked `is_swap=True` / wrong `swap_items` predicate

**Files modified:** `utils/telegram_bot.py`
**Commit:** `121ac04`
**Applied fix:** Added `item["is_swap"] = True` before the `STEP_SWAP_CONFIRM` transition in `_handle_serial_correction`. Simplified the `swap_items` list comprehension to use only `it["is_swap"]` (removing the fragile `or it["serial"] == corrected` branch that could double-count items).

---

### WR-01: `_next_index` called with wrong `base` argument — multi-serial photo naming corrupt

**Files modified:** `utils/telegram_bot.py`
**Commit:** `a8e8095`
**Applied fix:** Added `_strip_numeric_suffix()` helper function (using `re.sub(r"_\d{2}$", "", base)`) and applied it to compute `canonical_base` before calling `_next_index` in `_upload_and_reply`. This prevents double-indexed filenames like `01_Serial_Number_01_02.jpg`.

---

### WR-02: `classify_photo_names` always generates `01_Serial_Number_01.jpg` — diverges from convention

**Files modified:** `utils/telegram_bot.py`
**Commit:** `4464638`
**Applied fix:** Updated `classify_photo_names` so a single serial-label photo produces `01_Serial_Number.jpg` (no suffix). The `_01`, `_02`, ... suffixes are only applied when there are multiple serial labels in the same bookout, matching the convention used by `photos.py` and the CLI.

---

### WR-03: `lookup_site_type` is dead code — silently swallows all Drive exceptions

**Files modified:** `utils/telegram_bot.py`
**Commit:** `c998451`
**Applied fix:** Removed the dead `lookup_site_type` function entirely. All call sites already use `is_fmas_site()` from `utils/site_detection`. This also eliminates the bare `except Exception: return None` clause that would silently convert Drive API failures into a wrong site-type result.

---

### WR-04: Unit-number tokenization in `SiteStructureStore.learn` uses substring match

**Files modified:** `utils/telegram_state.py`
**Commit:** `07d5ba5`
**Applied fix:** Updated the tokenization loop in `SiteStructureStore.learn` to use exact-match (`seg == unit_number or seg == unit_folder_name`) as the primary branch, and restricted the substring branch to segments that start with `"Unit "`. This prevents a segment like `"Block 12"` from being corrupted to `"Block {unit}2"` when the unit number is `"1"`.
