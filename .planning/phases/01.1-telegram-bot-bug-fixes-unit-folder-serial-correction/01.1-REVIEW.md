# Phase 01.1: Code Review Report

**Reviewed:** 2026-04-16
**Depth:** standard
**Files Reviewed:** 2
**Status:** issues_found

## Summary

Both files implement the Telegram bot's one-shot bookout flow and its supporting
state management. The architecture is sound — media-group buffering, FSM-driven
conversation state, and the persistent site-structure store are all well-structured.

Two critical bugs exist in the serial-correction flow (`_handle_serial_correction`):
a corrected-but-still-not-found serial item never has its `is_swap` flag set to
`True` before the flow continues, which causes `update_stock_row` to be called with
a serial it will not find, raising an uncaught `ValueError` that kills the session
without a useful user-facing message.

Four warnings cover: a wrong base argument passed to `_next_index` during photo
conflict resolution, a naming-convention divergence for serial photos (always
`_01` suffix), dead code (`lookup_site_type`), and a fragile substring-based unit
tokenization in `SiteStructureStore.learn`.

## Critical Issues

### CR-01: Corrected-but-still-not-found serial never marked `is_swap=True` before stock update

**File:** `utils/telegram_bot.py:854-860`

**Issue:** When a technician replies with a corrected serial that is still not found
in any stock sheet, the handler sets `item["serial"] = corrected` (line 856) and
transitions to `STEP_SWAP_CONFIRM`, but **never sets `item["is_swap"] = True`**.
When the user then replies to the swap-confirm prompt, `_continue_after_swap_confirm`
iterates `state["items"]` and calls `update_stock_row` for every item where
`it["is_swap"]` is `False`. The corrected (but missing) serial is sent to
`update_stock_row`, which raises `ValueError("Serial number '...' not found in any
Serial Number Listing sheet.")`. This exception is caught by the outer `try/except`
in `_continue_after_swap_confirm` (line 766), clears state, and sends a generic
"Stock update failed" message — the bookout session is destroyed.

**Fix:**

```python
    else:
        # Still not found — fall back to swap confirm.
        item["serial"] = corrected        # keep corrected serial
        item["is_swap"] = True            # <-- ADD THIS LINE
        swap_items = [it for it in state["items"] if it["is_swap"]]
        state["step"] = STEP_SWAP_CONFIRM
        STATE.set(chat_id, state)
        await bot.send_message(chat_id, format_swap_warning(swap_items))
```

### CR-02: `swap_items` list in serial-correction fallback is built with incorrect predicate

**File:** `utils/telegram_bot.py:857`

**Issue:** The line that builds `swap_items` before entering `STEP_SWAP_CONFIRM` is:

```python
swap_items = [it for it in state["items"] if it["is_swap"] or it["serial"] == corrected]
```

At this point `item["serial"]` has already been overwritten to `corrected` (line 856),
so the `or it["serial"] == corrected` branch matches the current item regardless of
its `is_swap` flag. If a *different* item's serial happens to equal `corrected` (e.g.
a duplicate serial entry), that item is double-counted. More importantly, the current
item is included in the warning list not because it's marked as a swap but because
of the serial string match — when `is_swap` is eventually consulted by
`_continue_after_swap_confirm`, the item is treated as non-swap and the stock update
is attempted (see CR-01). After fixing CR-01, this predicate should be simplified to
use only the `is_swap` flag:

```python
swap_items = [it for it in state["items"] if it["is_swap"]]
```

## Warnings

### WR-01: `_next_index` called with a wrong `base` argument — multi-serial photo naming corrupt

**File:** `utils/telegram_bot.py:960-962`

**Issue:** In `_upload_and_reply`, when a photo's `want_name` already exists in the
folder, the conflict-resolution code does:

```python
base = want_name.rsplit(".", 1)[0]         # e.g. "01_Serial_Number_01"
final = want_name if want_name not in existing else _next_index(existing, base)
```

For a multi-serial bookout where `want_name` is `"01_Serial_Number_01.jpg"` and that
file already exists, `base` becomes `"01_Serial_Number_01"`. `_next_index` then looks
for names matching `01_Serial_Number_01_(\d+).jpg` and returns
`"01_Serial_Number_01_02.jpg"` — a double-indexed, non-standard filename. The
convention (per `photos.py`) uses `01_Serial_Number`, `01_Serial_Number_02`,
`01_Serial_Number_03` as its suffix sequence, not `01_Serial_Number_01_02`.

The `base` should be computed from the canonical prefix, not the per-message
`want_name`. For serial photos the canonical base is always `"01_Serial_Number"`;
for others it is the filename stem without the `_01` suffix.

One pragmatic fix is to strip a trailing `_NN` from `base` before calling
`_next_index`:

```python
import re as _re

def _strip_numeric_suffix(base: str) -> str:
    """Remove a trailing _01 / _02 etc from a base name."""
    return _re.sub(r"_\d{2}$", "", base)

# in _upload_and_reply:
base = want_name.rsplit(".", 1)[0]
canonical_base = _strip_numeric_suffix(base)
final = want_name if want_name not in existing else _next_index(existing, canonical_base)
```

### WR-02: `classify_photo_names` always generates `01_Serial_Number_01.jpg` — diverges from convention

**File:** `utils/telegram_bot.py:155-156`

**Issue:** For a single serial-label photo, `classify_photo_names` returns
`("serial", "01_Serial_Number_01.jpg")` (suffix `_01`). The canonical first filename
according to `photos.py` and the CLAUDE.md spec is `01_Serial_Number.jpg` (no suffix).
The suffix is only added when there are multiple serials in the same bookout, or
when a conflict exists with a previously uploaded file.

`_next_index(existing, "01_Serial_Number")` handles the conflict case internally —
it checks whether `01_Serial_Number.jpg` exists and only then applies `_02`, `_03`
suffixes. By pre-assigning `_01`, the bot bypasses this logic and always writes the
non-standard name, making the folder contents inconsistent with bookouts done via
the CLI or web app.

**Fix:**

```python
# In classify_photo_names, replace the loop:
if n_labels == 1:
    names.append(("serial", "01_Serial_Number.jpg"))
else:
    for i in range(n_labels):
        names.append(("serial", f"01_Serial_Number_{i + 1:02d}.jpg"))
```

### WR-03: `lookup_site_type` is dead code — silently swallows all Drive exceptions

**File:** `utils/telegram_bot.py:180-205`

**Issue:** `lookup_site_type` was defined to determine FMAS vs ATEC by inspecting
the Drive folder structure, but all call sites now use `is_fmas_site()` (the
static-list approach) instead. The function is never called. Additionally, its
`except Exception: return None` clause silently converts Drive API failures,
authentication errors, and network timeouts into a "site not found" result — which
would silently mis-route all bookouts to ATEC-direct mode during a Drive outage.

Either remove the function (it is unreachable) or, if it is intended to be wired
back in, replace the bare `except` with specific exception types and re-raise
unrecoverable errors:

```python
    except (FileNotFoundError, PermissionError):
        return None   # site genuinely absent from Drive
    # Do NOT catch googleapiclient.errors.HttpError here — let it propagate
```

### WR-04: Unit-number tokenization in `SiteStructureStore.learn` uses substring match

**File:** `utils/telegram_state.py:186-191`

**Issue:** The tokenization loop in `learn()` replaces any segment that *contains*
the unit number string with `{unit}` via `str.replace`:

```python
elif unit_number and unit_number in seg:
    template.append(seg.replace(unit_number, self.UNIT_TOKEN))
```

This is called from `_handle_nav_reply` (line 1079) where `segments` is the
breadcrumb path below `Sites/[site_name]/`. If a static path segment (e.g.
`"Block 12"`) happens to contain the same digit sequence as the unit number
(e.g. unit `"1"` and segment `"Block 12"`), the segment becomes `"Block {unit}2"`.
On a subsequent bookout for unit `"3"`, that resolves to `"Block 32"` — a
non-existent folder — silently creating it.

The exact-match branch (`seg == unit_number`) should be sufficient for the unit
folder itself. The substring branch should only apply when the segment is
*specifically* the unit folder name, which can be checked by testing whether the
segment equals the formatted unit name:

```python
unit_folder_name = f"Unit {unit_number}"
for seg in segments:
    if seg == unit_number or seg == unit_folder_name:
        template.append(self.UNIT_TOKEN)
    elif unit_number and unit_number in seg and seg.startswith("Unit "):
        # Only substitute within "Unit X"-style names, not arbitrary segments
        template.append(seg.replace(unit_number, self.UNIT_TOKEN))
    else:
        template.append(seg)
```

## Info

### IN-01: `_PROCESSED_GROUP_IDS` set grows without bound

**File:** `utils/telegram_bot.py:67, 583`

**Issue:** `_PROCESSED_GROUP_IDS` is a module-level `set[str]` that accumulates
every media group ID that has been flushed. Since media group IDs are never removed,
the set grows indefinitely for the lifetime of the process. On a busy deployment
(Railway, single instance) this is a slow memory leak. The set's purpose is to
prevent double-processing — a TTL-bounded structure (e.g. a deque capped at 1000
entries, or a dict with a timestamp eviction) would be more appropriate.

**Fix (minimal):** Replace with a capped deque check, or use a time-based eviction
keyed on media group ID. For current traffic volumes this is low-urgency but worth
addressing before scaling.

### IN-02: `cmd_checkstock` does not guard against `update.effective_user` being `None`

**File:** `utils/telegram_bot.py:1229`

**Issue:** `cmd_checkstock` calls `is_allowed(update.effective_user.id)` without
first checking that `update.effective_user` is not `None`. In `on_message` this
guard is present (line 462), but `cmd_checkstock` is registered as a
`CommandHandler` and could theoretically receive an update without a user (e.g.
channel post). If `update.effective_user` is `None`, the handler raises
`AttributeError` before any user-facing message is sent.

The same pattern applies to `cmd_cancel` (line 1262) and `cmd_start` (line 1273),
though those functions are less likely to be triggered from channels.

**Fix:**

```python
async def cmd_checkstock(update: Update, context: ContextTypes.DEFAULT_TYPE):
    if update.effective_user is None or not is_allowed(update.effective_user.id):
        return
```
